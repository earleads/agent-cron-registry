Setup guide

Ask your agent to schedule it. It shows up in one place.

A shared Notion table plus one script your AI coding agent calls, so every scheduled job across your team lands in the same place and actually runs, instead of scattered across five different tools.

Get the code Jump to setup

What this solves

Most teams end up with scheduled jobs scattered across whatever tool was closest at the time: one report lives in one automation platform, one check is a chat routine, one script is a cron job on someone's laptop nobody else knows exists. This gives every job one home: ask your agent for it in plain English, and it registers the job to a table everyone can see while installing the real job on the machine you asked it to run on.

Before you start

  • A Notion account. Free tier is fine. Everything else depends on this.
  • An always-on server. This is required, not optional. The whole point of this setup is that jobs keep running and stay visible to your whole team whether or not anyone's laptop is open. It doesn't need to be powerful or expensive, just always on. Mac or Linux only for now, see the Windows note below.
    Don't have one? Here's the 10 minute walkthrough to get one, from a few euros a month.
  • A chat-based coding agent like Claude Code. This is what makes the plain-English flow below work. Every script also runs fine by hand if you'd rather, so an agent isn't strictly required, but this guide is written assuming you have one.
  • GitHub CLI (gh), installed and logged in. This is the piece that lets you skip typing commands: with it in place, you just tell your agent "clone the repo" or "check for updates" and it handles the rest, no separate login step. Install it once from cli.github.com.
  • About fifteen minutes, most of it spent inside Notion setting up an account connection.

On Windows? This version schedules jobs with cron, which Windows doesn't have built in, so it only runs on Mac and Linux today. The simplest fix is the same always-on Linux server already required above, which also solves "keeps running when my laptop's closed" at the same time.

How to use this guide

Because you've got a coding agent and GitHub CLI set up, almost none of this needs the terminal. Each step below shows the exact words to say to your agent, in plain English. Two steps happen inside Notion's website and only you can do those (your agent can't log into your Notion for you), so they're clearly marked. Prefer running things yourself? The raw command is tucked under a "run it yourself" toggle on every step that has one.

Setup, step by step

No always-on server yet? Do that first, everything below assumes you have one. It takes about ten minutes and costs a few euros a month: follow the server walkthrough, then come back here.
1

Get the code

Bring the project onto the machine you'll be scheduling jobs on.

Say to your agent "Clone the agent-cron-registry repo from the earleads GitHub and open it up."
Run it yourself 
gh repo clone earleads/agent-cron-registry
cd agent-cron-registry
2

Create a Notion integration

You do this one

This gets you the token that lets the tool write to your Notion. It happens inside Notion's website, so your agent can't do it for you.

  1. Go to notion.so/my-integrations, logged into the Notion account you want to use.
  2. Click "New integration" and give it any name, for example "Cron Registry".
  3. Copy the "Internal Integration Token" it shows you. It's a long string starting with secret_ or ntn_. Treat it like a password and keep it handy for step 4.
3

Connect the integration to a page

You do this one

This tells Notion which page the new table is allowed to live under. Also inside Notion's website, so it's on you.

  1. Open or create any page you're happy to have a new table live under.
  2. Click the "..." menu in the top right, choose "Connections", and pick the integration you just made.
  3. Copy that page's ID: it's the long string of letters and numbers at the end of the page's web address. You'll hand this to your agent in step 5.
4

Hand over your token

Give your agent the token from step 2 so the tool can talk to Notion.

Say to your agent "Set up the environment file for this project and save my Notion token in it: [paste the token from step 2]."
Run it yourself 
cp .env.example .env
# then open .env and paste your token in as:
# NOTION_API_TOKEN=your_token_from_step_2
# leave NOTION_REGISTRY_DB_ID blank for now
5

Create the shared table

Now your agent builds the "Scheduled Jobs" table in Notion and wires the last ID back in for you, so there's nothing left to copy-paste.

Say to your agent "Use this Notion page ID to create the Scheduled Jobs table, then save the database ID it gives back into my environment file: [paste the page ID from step 3]."
Run it yourself 
./scripts/setup-notion-db.sh <page-id-from-step-3>
# it prints a database ID, paste that into .env as:
# NOTION_REGISTRY_DB_ID=the_printed_id
6

Try it

Schedule a throwaway job to prove the whole thing works end to end.

Say to your agent "Schedule a test job that writes the word hello to a file every five minutes, so we can check this works."

Then check two things: a new row should appear in your Notion table right away, and after five minutes the hello file should have a line in it. Both happening means everything is wired up. Once you've confirmed it, clean up:

Say to your agent "Great, remove that test job, both the schedule and the Notion row."
Run it yourself 
./scripts/register-job.sh "Say hello" "*/5 * * * *" "Every 5 minutes" "echo hello >> /tmp/hello.log"
# to remove it afterwards:
crontab -l | grep -v 'Say hello' | crontab -
# then delete or pause the row in Notion
7

Let your agent handle scheduling from now on (optional)

Install the little skill that teaches your agent this whole flow, so from here on you just ask for jobs in plain English and it does the rest.

Say to your agent "Install the schedule-job skill from this repo into my project so I can just ask you to schedule things."

After that, requests like "schedule my morning report every weekday at 8am" just work: your agent figures out the timing and the command and registers the job itself.

Run it yourself 
cp -r claude-skill/schedule-job /path/to/your/project/.claude/skills/schedule-job
8

Add the guardrail (optional but recommended)

This stops your agent from quietly installing a job that skips the registry, which matters once more than one person is asking it to schedule things.

Say to your agent "Add the guardrail hook from this repo that blocks any cron job which skips the registry, and wire it into my project's settings."
Run it yourself

The exact .claude/settings.json snippet is in docs/SETUP_GUIDE.md in the repo.

Want the raw copy-paste commands for every step in one place, plus troubleshooting for common errors? They live in docs/SETUP_GUIDE.md in the repo.

What this doesn't do yet

  • No web dashboard, the Notion table is the only view.
  • No one-click removal yet. To take a job out you (or your agent, if you ask it) edit the crontab and the Notion row.
  • Only tested against standard cron. Works the same with launchd or GitHub Actions instead, you'd just install the job a different way in step 6.
